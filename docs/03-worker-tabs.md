# 03. 작업자 탭 가이드

이 문서는 SwarmUI의 각 탭을 무엇에 쓰는지 정리한다.  
여기서 목표는 워크플로를 만드는 법이 아니라, 작업자와 운영자가 어떤 화면을 언제 써야 하는지 분리해서 이해하는 것이다.

## 1. 먼저 기억할 것

SwarmUI에서 대부분의 작업자는 `Generate`와 `Simple`만 알아도 된다.  
`Comfy Workflow`는 워크플로 작성자용이고, `Utilities`와 `Server`는 운영자 비중이 크다.

작업자 기준으로 가장 중요한 원칙은 아래 네 가지다.

1. 작업자는 준비된 UI를 사용한다.
2. 그래프가 깨지거나 노드가 없어 보이면 직접 고치지 않는다.
3. 모델이 안 보이거나 결과가 전부 실패하면 운영자에게 넘긴다.
4. 업무 중에는 `Generate` 또는 `Simple`에서 재현 가능한 방식으로 작업한다.

## 1-1. 작업 시작 전 확인

작업자는 매번 아래 순서로 시작하는 것이 좋다.

1. `SwarmUI` 접속 주소가 맞는지 확인한다.
2. 본인 계정으로 로그인한다.
3. 오늘 사용할 모델이 목록에 보이는지 확인한다.
4. 이전 작업 히스토리와 출력 경로가 정상인지 확인한다.
5. 필요한 경우 팀에서 정해 둔 `Preset`이나 `Simple` 템플릿을 먼저 불러온다.

이 단계에서 문제가 있으면 바로 운영자에게 넘긴다.

- 로그인 불가
- 모델 목록 비정상
- 화면이 비어 있음
- 결과 저장 위치가 이상함
- 생성 버튼이 눌러도 반응 없음

## 2. `Simple` 탭

`Simple` 탭은 미리 준비된 워크플로를 단순한 입력 화면으로 보여주는 방식이다.  
워크플로 작성자가 `SwarmInput` 계열 입력을 잘 정리해 두면, 작업자는 복잡한 노드 그래프를 보지 않고도 필요한 값만 넣을 수 있다.

- 이런 탭은 반복 업무를 맡는 작업자에게 좋다.
- 워크플로를 이미 정한 팀에서 특히 유용하다.
- 작업자는 그래프를 고칠 필요가 없다.
- 작성자는 어떤 값만 노출할지 설계해야 한다.

### 2-1. `Simple` 탭에서 작업하는 순서

1. 팀에서 배포한 템플릿이나 워크플로를 선택한다.
2. 노출된 입력값만 채운다.
3. 샘플 한 건을 먼저 돌린다.
4. 결과가 맞으면 필요한 수량만큼 반복한다.
5. 결과물을 확인하고 저장 경로를 점검한다.

### 2-2. `Simple` 탭에서 작업자가 하지 말아야 할 것

- 보이지 않는 파라미터를 억지로 바꾸려고 하지 않는다.
- 템플릿 구조가 마음에 안 든다고 그래프 수정을 시도하지 않는다.
- 노드가 누락된 상태를 직접 복구하려고 하지 않는다.
- 결과 형식이 이상하다고 `Comfy Workflow`로 넘어가서 임의 수정하지 않는다.

이런 경우는 워크플로 작성자 또는 운영자 이슈다.

## 3. `Generate` 탭

`Generate` 탭은 대부분의 작업자가 매일 쓰는 메인 화면이다.  
모델을 고르고, 프롬프트를 넣고, 파라미터를 조정하고, 큐를 돌리고, 결과를 확인하는 곳이다.

주로 하는 일은 다음과 같다.

- `Models`에서 사용할 모델을 고른다.
- 프롬프트와 네거티브 프롬프트를 넣는다.
- 샘플링, 해상도, 시드 같은 기본 값을 조정한다.
- `Grid`나 배치로 조건을 비교한다.
- 이미지 히스토리와 출력 결과를 확인한다.

Generate 탭은 워크플로 그래프를 직접 만드는 곳이 아니다.  
작업자 입장에서는 여기서 빠르게 반복하고, 결과가 좋은 설정을 찾는 것이 핵심이다.

### Generate 탭에서 자주 쓰는 것

- `Models` 브라우저: 모델 선택과 메타데이터 확인
- `Prompt` 입력: 생성 요청
- `Presets`: 자주 쓰는 파라미터 묶음 저장
- `Wildcards`와 프롬프트 문법: 반복 작업 자동화
- 이미지/출력 히스토리: 결과 검토와 재사용

참고: `Grid Generator`는 `Utilities` 탭에 있다. 여러 조건을 비교할 때 쓰지만 Generate 탭 화면 안에는 없다. 자세한 사용법은 [04. 운영 도구와 API](04-admin-operations.md)에서 본다.

### 3-1. `Generate` 탭 기본 작업 절차

1. `Models`에서 승인된 모델을 고른다.
2. 프롬프트와 네거티브 프롬프트를 입력한다.
3. 해상도, 시드, 샘플링 기본값을 확인한다.
4. 1장만 먼저 생성한다.
5. 결과가 맞으면 배치 수량이나 반복 작업으로 늘린다.
6. 출력물과 히스토리를 확인한다.

### 3-2. 모델 선택 시 작업자 기준

작업자는 아래 기준만 지키면 된다.

- 팀에서 허용한 모델만 쓴다.
- 모델명이 비슷해도 임의로 새 모델을 선택하지 않는다.
- 타입이 이상하게 보이면 직접 메타데이터를 고치지 않는다.
- 모델이 안 보이면 운영자에게 알려서 경로/메타데이터를 확인하게 한다.

### 3-3. `Generate` 탭에서 자주 보는 영역

- 상단 프롬프트 영역: 생성 요청 입력
- 모델 선택 영역: 현재 모델과 모델 브라우저
- 파라미터 그룹: 해상도, 샘플링, 시드, 고급 옵션
- 대기열/진행 상태: 현재 생성 상태 확인
- 결과 영역: 생성물 미리보기
- 히스토리/출력 영역: 이전 결과 다시 보기

### 3-4. 작업자가 운영자에게 넘겨야 하는 상황

아래 상황은 작업자 선에서 해결하지 않는다.

- 같은 프롬프트로 계속 실패한다.
- 특정 모델만 고르면 오류가 난다.
- 큐가 계속 멈춘다.
- 생성은 되는데 저장이 되지 않는다.
- UI는 열리는데 `Backends` 문제로 보이는 에러가 난다.

## 4. `Comfy Workflow` 탭

`Comfy Workflow` 탭은 raw ComfyUI 그래프를 직접 만지는 곳이다.  
이 탭은 작업자용이 아니라 워크플로 작성자와 파워 유저용이다.

이 탭에서 하는 일은 다음과 같다.

- 노드 그래프를 직접 만든다.
- 외부에서 가져온 Comfy 워크플로를 불러온다.
- 생성용 구조를 검토한다.
- `Generate` 탭에 연결할 워크플로를 준비한다.

일반 작업자는 이 탭을 열어볼 수는 있어도, 직접 고칠 책임은 없다.  
작성자가 완료한 워크플로를 사용하는 게 원칙이다.

### 4-1. 작업자가 알아야 할 최소한의 원칙

- 이 탭은 “사용”보다 “설계/검토”를 위한 탭이다.
- 실수로 들어갔다면 임의 저장하지 말고 빠져나온다.
- 팀에서 특별히 지정한 경우가 아니면 `Use This Workflow`를 누르지 않는다.
- 노드 누락, 빨간 박스, 로드 실패는 바로 보고한다.

## 5. 이 문서에서 다루지 않는 것

`Utilities`, `User`, `Server`, `API`, 모델 다운로드 같은 운영 메뉴는 이 문서의 중심이 아니다.  
그 항목들은 [04. 운영 도구와 API](04-admin-operations.md)에서 따로 본다.

## 6. 작업자가 맡고, 맡지 않는 것

작업자가 맡는 것.

- `Generate`에서 생성하기
- `Simple`에서 준비된 인터페이스 쓰기
- 결과 확인과 재시도
- 개인 설정 조정
- 필요할 때만 공유된 프리셋 사용

작업자가 보통 맡지 않는 것.

- 새 워크플로 그래프 만들기
- 백엔드 추가와 제거
- 서버 설정과 포트 변경
- 사용자 역할 변경
- 확장 설치
- 모델 경로 구조 설계

## 7. 작업자 인수인계 체크리스트

새 작업자에게 계정을 넘길 때 아래를 확인한다.

- 접속 URL을 알고 있다.
- 본인 계정으로 로그인할 수 있다.
- `Simple`과 `Generate`의 차이를 이해했다.
- `Comfy Workflow`를 직접 수정하지 않는다는 점을 이해했다.
- 오류 발생 시 어디까지가 작업자 책임인지 안다.
- 결과물이 저장되는 위치를 안다.

첫 출근 또는 첫 배포 후에는 아래 테스트까지 실제로 해보는 편이 좋다.

1. 로그인한다.
2. 팀에서 지정한 모델 1개가 목록에 보이는지 확인한다.
3. `Generate` 또는 `Simple`에서 테스트 작업 1건을 보낸다.
4. 결과가 열리는지 확인한다.
5. 저장 위치나 히스토리에서 결과를 다시 찾을 수 있는지 확인한다.
6. 문제가 생기면 누구에게 넘길지 알고 있는지 확인한다.
